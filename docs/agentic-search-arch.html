JS Agent Architecture

Browser-native multi-step agent with modular app/runtime layers, hosted and local model routing, browser skills, filesystem fallbacks, notifications, and multi-tab coordination.

runtime: browser
loop: max rounds + ctx budget
skills: registry-based
multi-tab: BroadcastChannel

Current Runtime Snapshot

UI shell
Collapsible sidebar + retractable panels
Chat, model controls, runtime toggles, sessions, capability groups, and activity live in one browser page.
Agent loop
Round-bounded tool loop
The app builds a system prompt, sends message history to a model, executes at most one tool per turn, and injects tool results back into context.
Routing
Hosted or local model path
Requests can go to Gemini or to a locally probed OpenAI-compatible backend such as LM Studio or Ollama-style endpoints. Best local results so far: mistralai/ministral-3-3b.
Storage
Browser-only persistence
API key, sessions, active session id, local backend preferences, sidebar state, and tool cache are kept in localStorage.

Execution Flow

1. Bootstrap and module wiring `index.html` loads core, skills, and app modules in order.
+

The page loads prompt helpers, skill modules, the orchestrator, and then the app modules. `DOMContentLoaded` restores sessions, initializes sidebar state, probes local backends, and wires tab-level sync channels.

<script src="src/core/regex.js" defer></script>
<script src="src/core/prompt-loader.js" defer></script>
<script src="src/skills/shared.js" defer></script>
<script src="src/core/orchestrator.js" defer></script>
<script src="src/app/state.js" defer></script>
<script src="src/app/agent.js" defer></script>
2. Prompt creation and policy layer `prompt-loader.js` + `orchestrator.js` build the final system prompt.
+

The orchestrator receives the list of enabled tools and composes the prompt contract. It also owns tool-call parsing, reasoning-leak repair prompts, and summary prompts when context gets too large.

buildSystemPrompt({
  userMessage,
  maxRounds,
  ctxLimit,
  enabledTools
})
3. Agent loop The app keeps iterating until a final answer or max rounds.
+

The loop sends current history to the selected model, parses one tool call if present, executes it through the registry, appends a `tool_result` block, and continues. If the model answers directly, the loop stops and renders a Markdown-first response through the sanitizer pipeline.

while (round < MAX_ROUNDS) {
  const reply = await callModel(messages);
  const toolCall = parseToolCall(reply);
  if (!toolCall) return finalAnswer(reply);

  const result = await executeTool(toolCall);
  messages.push({ role: "user", content: toolResultBlock(toolCall, result) });

  if (ctxSize(messages) > CTX_LIMIT) {
    messages = await summarizeContext(userMessage);
  }
}
4. Skill execution Tools resolve through `window.AgentSkills.registry`.
+

The registry currently covers search, parsing, clipboard, storage, notifications, multi-tab messaging, and filesystem actions. The app can toggle tool availability without changing skill code.

{
  web_search,
  read_page,
  parse_json,
  clipboard_write,
  notification_send,
  tab_broadcast,
  tab_listen,
  fs_write_file
}

Important Behaviors

  • Filesystem writes prefer direct File System Access when available and fall back to browser download when not.
  • Notification permission is requested from user-triggered actions such as saving the API key or probing the local backend.
  • When the page is hidden, the app can fire a native notification after a run finishes.
  • Multiple tabs share tool-cache entries over `BroadcastChannel`, reducing repeated work for identical calls.
  • Tabs also broadcast busy state so another window can detect that a run is already in progress.
  • The agent itself can publish and listen for messages across tabs through `tab_broadcast` and `tab_listen`.
  • For file-oriented tasks, guardrails force root exploration (`fs_list_dir('/')`) before direct file reads to reduce invalid-path loops.

Module Map

src/app/state.js
Persistent state and shared UI/runtime flags
API key, sessions, sidebar state, enabled tools, tool cache, cache sync, and busy sync live here.
src/app/agent.js
Loop execution and chat rendering
Owns `sendMessage()`, `agentLoop()`, status updates, final rendering, and completion notifications.
src/app/tools.js
UI-facing tool controls and prompt wiring
Renders tool groups, toggles enabled tools, and delegates prompt construction to the orchestrator.
src/app/local-backend.js
Local endpoint detection
Probes known localhost candidates, detects model lists, and activates local routing when available.
src/core/orchestrator.js
Policy and parsing
Builds prompts, validates skill availability, parses tool calls, and generates repair and summary prompts.
src/skills/shared.js
Primary skill runtime
Holds helper functions, preflight heuristics, device/browser skills, and the registry exposed as `window.AgentSkills`.

Multi-Tab Architecture

Case A
Agent-triggered tab messaging
The model can call `tab_broadcast(topic, payload)` to publish a result and `tab_listen(topic, timeout_ms)` to wait for work happening in another tab.
Case B
App-internal tab sync
Without involving the model, tabs sync tool-cache entries and busy state through dedicated `BroadcastChannel` names.
Scope
Same origin only
This works only across tabs/windows using the same origin and browser profile. It does not cross devices or different browsers.
BroadcastChannel channels

loopagent-v1
  Agent-level publish/listen tools for cross-tab coordination

loopagent-cache-v1
  Internal cache replication for tool results

loopagent-busy-v1
  Internal "another tab is running" state
